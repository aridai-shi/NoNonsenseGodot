Introduction to Scenes and Nodes

Nodes are the basic building blocks of a Godot project. Different parts of the game world will often be made up of several different Nodes. Some will be responsible for collision, others for visual effects, physics handling, etc. Their class (ex. KinematicBody2D, TextureProgress, CollisionShape) will determine their built-in functions and properties, streamlining programming by using the right Node for the right job (ex. KinematicBody2D has a move_and_slide() function that lets' it move around while accounting for collision). Nodes are ordered in a tree structure - when Node A is put under Node B in the structure, we say Node A is Node B's child Node. Child Nodes inherit some of their properties from their parent Nodes (ex. position, rotation, scale), only being able to change them in relation to their parents.

In the left picture, Node B is Node A's sibling. In the right pic Node B is Node A's child. Know the difference!

Child Nodes are also used by parent Nodes as components (ex. an Area2D Node using a CollisionShape2D child Node for collision). To use a child Node within the parent's script, use $ChildNodeName.

EXAMPLE: To mess with the HandSprite Node from Player's script, you gotta call $HandArea/HandSprite.

Scenes are just Nodes saved as files. You can copy-paste the Scene all around a project, and when you change the Scene's file directly, the changes will be applied to all instances of said Scene immediately.

EXAMPLE: By moving the CollisionShape2D in Player.tscn, we've affected all versions of it throughout the game.

The ability to make tons of copies of different Node setups without clogging up the tree and to change the object once instead of manually changing each copy makes Scenes an essential tool for any Godot game developer.
